1. Vision general: 3 caminos de analisis

El sistema tiene 3 paths independientes que analizan las notas/contenido del deal de formas diferentes:

PathQue haceDonde correCuando se ejecutaOutput
A Module Analysis Mapea contenido a modulos de Factorial Cloudflare Worker /ai Automatico al cargar datos en StepSetup ModuleSuggestion[] con confidence strong/possible
B Pain Mapping Mapea notas a pains de la pain_library Supabase Edge Function Manual desde StepAiAssist o inline en Airtable fetch {pain_id, rationale}[]
C Evidence Analysis Extrae evidencia + matchea a modulos con reglas DB Supabase Edge Function (2 llamadas a Azure) Manual: boton "Download evidence analysis" Excel con 4 hojas (evidence, matches, summary, meta)
Diferencia clave: Modulos vs Pains

Modulos = productos de Factorial (Time Tracking, Payroll, Performance...) definidos en MODULE_CATALOG en el frontend.
Pains = problemas especificos del prospect por persona (HR Director, Finance, etc.) definidos en pain_library en Supabase.
Path A y C trabajan con modulos. Path B trabaja con pains. Son taxonomias diferentes usadas en pasos diferentes del wizard.

3. Path A: Analisis de modulos

PATH A StepSetup.tsx → Cloudflare Worker /ai → Azure Claude
buildDealContent(data)
→
POST Worker /ai
→
Azure Claude claude-opus-4-6 (temp=0)
→
ModuleSuggestion[]

System prompt (resumen)

  • ~130 lineas de instrucciones detalladas
  • Prioridad: calls > incoming emails > outgoing emails > notes
  • Solo detectar pains del prospect, NO del seller (ignorar pitch del vendedor)
  • Confidence: strong = evidencia directa del cliente, possible = inferido o indirecto
  • Cada modulo incluye quote o ~parafraseo si no hay cita exacta
  • Incluye catalogo completo de modulos via buildModulePromptBlock()

Tool forzado: recommend_modules

{
  recommendations: [{
    module_id: string,     // ej: "time_tracking"
    confidence: "strong" | "possible",
    quote: string          // cita del prospect o ~parafraseo
  }]
}

Post-procesamiento

  1. Filtra modulos con IDs invalidos (no estan en MODULE_CATALOG)
  2. Deduplica por module_id
  3. Solo pre-selecciona los strong
  4. Los possible se muestran pero sin check

4. Path B: Pain Mapping

PATH B StepAiAssist.tsx → Supabase ai-assist-pain-mapping → Azure Claude / Gemini fallback
Notas manuales o HubSpot notes
→
supabase.functions.invoke("ai-assist-pain-mapping")
↓
Fetch pain_library (pain_id, persona, pain_statement, trigger_phrases)
→
Azure Claude (tool: suggest_pains)
→
{pain_id, rationale}[]
Si Azure falla (429/timeout) → fallback a Gemini 2.5 Flash via Lovable Gateway

Matching logic

  • Usa trigger_phrases de cada pain en la pain_library como senal de matching
  • El AI busca frases, sintomas o situaciones en las notas que coincidan con los triggers
  • Solo devuelve pain_ids validos (filtrado post-AI)
  • Rationale en el idioma del prospect

Retry logic (Azure 429)

  • Hasta 3 intentos
  • Parsea "wait N seconds" del body del 429
  • Espera min(N, 60) segundos entre intentos
  • Si Azure falla 3 veces → fallback a Gemini

Paso 1: Tipos de evidencia

evidence_typeDescripcionEjemplo
direct_quotePalabras exactas del cliente describiendo un problema"Tardamos 3 horas cada semana en cuadrar los horarios"
metricNumero o medida cuantificable"tenemos 200 empleados y 3 personas en RRHH"
situationWorkflow, proceso o escenario descrito"Los managers envian las vacaciones por email"
tool_mentionHerramienta o sistema especifico que usan"Estamos usando Excel para el control horario"
sentimentLenguaje emocional o frustracion"Estamos hartos de hacer todo manual"
process_gapCapacidad que falta o workaround"No tenemos forma de medir el rendimiento"

Paso 1: Clasificacion de atribucion

attributionCuando se usaMultiplicador de confianza
client_verbatimCita exacta del cliente (de call o email entrante)1.0
client_paraphraseParafraseo cercano de lo que dijo el cliente0.85
pae_interpretationInterpretacion del vendedor (email saliente, nota)0.65
inferredInferido del contexto, no dicho explicitamente0.4

Paso 2: Reglas de DB

La tabla evidence_rules contiene ~65 reglas como:

[R12] module=time_tracking | type=direct_quote | strength=strong | pattern="control horario manual"
[R13] module=time_tracking | type=tool_mention | strength=moderate | pattern="Excel para fichajes"
[R14] module=time_tracking | type=process_gap  | strength=strong | pattern="no registran horas"

La tabla module_evidence_profiles define senales por modulo:

time_tracking:
  strong_signals: ["fichaje manual", "control horario", "horas extra no registradas"]
  moderate_signals: ["flexibilidad horaria", "turnos"]
  anti_signals: ["ya tienen fichaje digital", "contentos con su sistema actual"]

6. Sistema de scoring deterministico

El scoring se calcula en codigo, no por la IA. Esto garantiza consistencia.

Formula

confidence = max(strength_base * attribution_mult) + (num_evidence - 1) * 0.05
// capped at 1.0
Strong + Client Verbatim
0.90
Strong + Paraphrase
0.77
Moderate + Verbatim
0.60
Moderate + PAE Interp.
0.39
Weak + Inferred
0.12
Bonus por evidencia extra
+0.05

Niveles de confianza

NivelRangoSignificado
HIGH≥ 0.70Evidencia fuerte directa del cliente
MEDIUM≥ 0.40Evidencia moderada o indirecta
LOW< 0.40Solo inferencias o interpretaciones del vendedor

7. Output de cada path

Path A: Module suggestions (lo que ve el usuario)

{
  module_id: "time_tracking",
  confidence: "strong",           // o "possible"
  quote: "Tardamos 3h semanales en cuadrar horarios"
}

Esto se muestra en StepSetup como tarjetas de modulos. Los strong se pre-seleccionan.

Path B: Pain suggestions

{
  pain_id: "P_TT_01",
  rationale: "El prospect menciona que no tienen control de horas"
}

Se usa en StepAiAssist/StepPains para pre-seleccionar pains en el wizard.

8. Limitaciones conocidas

1 Contenido NO se persiste

Emails, calls y notes solo viven en React state. Si el usuario recarga la pagina, se pierden. Hay que re-fetchear del deal. Los moduleSuggestions y selectedModules SI se guardan en Supabase.

2 Sin contenido = sin analisis

buildDealContent() requiere al menos emails, calls o notes. Si el deal no tiene ninguno (o el fetch fallo), los Path A y C devuelven error. No hay textarea de notas manuales en StepSetup.

3 Truncamiento de contenido

Calls: max 4,000 chars. Incoming emails: max 800 chars. Outgoing: max 400 chars. Notes: max 800 chars. Informacion importante al final de transcripts largos se puede perder.

4 Pais limitado a ES/FR

El tipo de country solo acepta "ES" o "FR". Cualquier otro pais se mapea a null y usa ES por defecto.

5 Evidence analysis NO actualiza el wizard

El Path C (evidence analysis) descarga un Excel pero NO escribe los resultados de vuelta en el wizard. Es una herramienta de auditoria standalone. No influye en la seleccion de modulos.

6 Fallback a modelo diferente

Si Azure falla en Path B, se usa Gemini 2.5 Flash (via Lovable). Es un modelo completamente diferente que puede dar resultados de calidad distinta. Path A y C NO tienen fallback - si Azure falla, fallan.

7 Path A sin retry automatico

El Cloudflare Worker (Path A) no tiene la logica de retry 429 visible desde el frontend. Si falla, el usuario tiene que pulsar "Reanalyze" manualmente. Los edge functions de Supabase (Path B y C) SI tienen retry.

8 IDs de Airtable hardcodeados

Los field IDs de Airtable estan hardcodeados en airtable-deal-fetch/index.ts. Si cambia el schema de Airtable, el fetch devuelve campos vacios silenciosamente.

9 HubSpot notes contienen HTML

Se limpian con regex replace(/<[^>]*>/g, "") que puede fallar con HTML malformado. Podria pasar contenido con tags rotos al AI.

10 RLS permisivo en evidence_matches

Las politicas RLS de evidence_matches usan true para INSERT y SELECT. Cualquier usuario autenticado puede leer/escribir matches de cualquier sesion.
